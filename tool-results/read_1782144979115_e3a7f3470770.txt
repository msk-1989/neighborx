    12→  - Hardcoded port 3003 (no PORT env).

    16→  - `message` handler accepts `{ roomId, senderId, senderName, text }`, attaches `id` (crypto.randomUUID()) and `createdAt` (ISO timestamp), and broadcasts to everyone in the room INCLUDING the sender via `io.to(room).emit("message", ...)`.
    17→  - On disconnect: decrements room counter and emits updated `presence`.
    18→  - Graceful SIGINT/SIGTERM shutdown.
    19→- Ran `bun install` in the service dir (resolved socket.io@4.8.3, @types/socket.io@3.0.2, @types/bun@1.3.14).

    21→- Verified via curl: `GET http://localhost:3003/` -> `{"ok":true,"service":"neighborx-chat"}` (HTTP 200).
    22→- Verified socket.io engine.io polling handshake on `/socket.io/?EIO=4&transport=polling` returns a valid `sid`.

    64→Work Log:

    66→- Root cause identified: Zustand `persist` middleware in `src/lib/store.ts` was rehydrating synchronously from localStorage BEFORE React hydration. If the user had previously navigated to a non-dashboard module, the client's first render used a different `activeModule` than the server, shifting the React tree and all `useId()` allocations.
    67→- Applied fix in `src/lib/store.ts`:
    68→  - Added `skipHydration: true` to the persist config to prevent automatic synchronous rehydration.
    69→  - Added `partialize: (state) => ({ neighborhood: state.neighborhood })` to only persist user preferences (scope), never navigation state (activeModule).
    70→- Applied fix in `src/components/nx/app-shell.tsx`:
    71→  - Added `React.useEffect(() => { void useNX.persist.rehydrate(); }, [])` to manually rehydrate persisted preferences AFTER hydration completes, so the client's first paint matches the server HTML exactly.
    72→- Applied fix in `src/app/layout.tsx`:
    73→  - Added `suppressHydrationWarning` to the `<body>` tag as a safety net for next-themes class manipulation.
